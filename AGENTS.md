# AGENTS.md

## Scope
- This repo is a **.NET 8** solution (`FairShareMonApi.sln`) with one Web API app (`FairShareMonApi`).
- Domain: a personal/group expense ledger & debt-splitting API ("Sổ ghi nợ chi tiêu"). Every resource is owned by a user (`user_id` scoping). Feature spec (features/use cases/business rules only): **`The-ideal.md`**; old technical spec at commit `6b19f01`.
- Conventions adapted from the sibling **quick-ordering** project; see `CLAUDE.md` → ".NET 8 adaptations" for what differs.

## Architecture Map (read these first)
- Bootstrap + DI + middleware pipeline: `FairShareMonApi/Program.cs`.
- Attribute-driven DI scanning via the **DiDecoration** package (`[ScopedService]`, `[SingletonService]`, `[TransientService]`).
- API shape: `AppController` enforces wrapped responses via `[ResponseWrapped]` → `ApiResult<T>`.
- Request flow: `Controllers/* -> Services/Api/* -> Repositories/* -> AppDbContext`.
- Data access helpers: `Repositories/Abstractions/BaseRepository.cs` (`ExecuteQueryAsync`, `ExecuteTransactionAsync`, `Query<T>` with AsNoTracking + soft-delete).
- Entities: `Database/Entities/<Name>.cs` (POCO `partial`) + `Database/Entities/Partials/<Name>.cs` (ctor + static `ConfigureModel(ModelBuilder)`).

## Domain Flows (feature spec: The-ideal.md; mechanics are repo conventions)
- **Auth:** opaque stateful token — login stores SHA-256(token) in `auth_tokens` + Redis (TTL = expiry); access + refresh pair; logout/password-change revokes from whitelist.
- **Expense + shares:** created together in one transaction. Default payer = the owner member (`is_owner`); a creator share with `amount = 0` is auto-inserted.
- **Debt balance:** computed **within one event only**; per member, `balance = paid (sum of expenses where member is payer) − owed (sum of own shares)`. Positive = owed to them; negative = they owe. Loose expenses use the settled flag instead.
- **Event lifecycle:** `OPEN`/`CLOSED`; CLOSED disables all writes to expenses in the event (sole exception: settled flag), is one-way, and never happens automatically. Expense `expense_time` must fall within the event's `[start_date, end_date]`.

## Project-Specific Conventions
- Prefer service attributes over manual DI registration. For multiple impls of one interface set `Multiple = true`, else `TryAdd` drops later ones.
- Use `ExecuteTransactionAsync` + `TransactionContext.NoCommit()` for validation/business failures inside writes.
- Soft-delete is standard (`members.is_active`, and/or `IsDeleted`); excluded by default in `BaseRepository.Query`. List endpoints expose `?include_inactive=true` for stats/export.
- **Resource Owned:** scope every owned-resource query by `user_id`; return 404 (not 403) on miss. Validate cross-user FK links (`payer_member_id`, share `member_id`).
- Money: `DECIMAL`/`decimal` or integer smallest-unit; never float. `amount >= 0` enforced by DB CHECK.
- IDs: `ulong Id` (bigint unsigned) PK **plus** a separate `Uuid` string column generated by the `Uuid.NewV7()` helper (UUIDv7, .NET 8-compatible — do not use `Guid.CreateVersion7()`).
- Lists support pagination `?limit=&offset=` (default 20, max 100).

## External Integrations and Boundaries
- DB: MySQL/MariaDB via Pomelo (`ConnectionStrings:Default`).
- Cache: Redis via StackExchange.Redis (token whitelist + TTL; runs locally under Docker).
- Auth: opaque stateful token scheme (hashed in DB + cache).

## Process Rules (important)
- **Clarification-First:** never assume; ask the user when anything is ambiguous, missing, or preference-dependent.
- **Planning doc before code:** create `/planning/[main-purpose].md` (lowercase kebab-case) before implementing; keep it synced; finish it before marking a task done.
- **DB Change Rule:** schema changes use EF Core migrations: `dotnet ef migrations add <Name> --project .\FairShareMonApi\FairShareMonApi.csproj` then `dotnet ef database update --project .\FairShareMonApi\FairShareMonApi.csproj`. Authoring commands run offline via `AppDbContextDesignTimeFactory` (pinned MariaDB `ServerVersion` — bump it if the target server major/minor changes); review the generated migration before applying. No manual `database-migration.sql`.
